>>
TaskGroup and SubDAG are both used to organize complex DAGs, but they solve very different problems.
SubDAG creates a separate DAG inside a parent DAG. It has its own scheduler, start date, and execution context. often caused scheduler delays, deadlocks, and performance issues.
TaskGroup - logically group related tasks within the same DAG, helps to create parallel processing of etl.
👉 Today, TaskGroup is the recommended approach, and SubDAG is considered anti-pattern / deprecated in practice.

sensors: A file is uploaded to Google Cloud Storage, and the pipeline should start only after the file arrives.
triggers:  After data ingestion DAG finishes, it triggers a data transformation DAG.

    wait_for_file = GCSObjectExistenceSensor(
        task_id="wait_for_file",
        bucket="data-bucket",
        object="input/customers.csv",
        poke_interval=30,
        timeout=600,
        mode="reschedule" or "poke" / deferrable=True
    )

poke → worker continuously occupied
reschedule → worker released between checks
deferrable → async trigger handles waiting, no worker usage

>> Worker occupied entire time (sensors - poke mode)
Worker starts task
↓
Sensor checks condition
↓
Sleep for poke_interval
↓
Check again
↓
Repeat until success or timeout

>> Worker used briefly Then released
Worker checks condition
↓
If false → task paused
↓
Worker released
↓
Scheduler wakes task later
↓
Condition checked again

>> Almost zero worker usage
Worker starts sensor
↓
Registers async trigger
↓
Worker exits
↓
Triggerer process monitors condition
↓
When event occurs → task resumes
